Act 2

5 PM Beliefs
Upended

Title echoes the deck name "The Upended Product Cycle." Format inspired by Anthropic's "Beliefs for Engineers" and Every.to's "Compound Engineering" - mention only if asked about the structure.
Belief #1
"Front-load all research before building"

Building was expensive. One wrong bet could waste a quarter. So we front-loaded certainty: weeks of research before greenlighting anything.

When a prototype takes hours, not months, you build to learn, not learn to build.

Before

Discover
→
Deliver

Sequential. Weeks apart.

Now

Build
Learn
Build
Learn

↻ Continuous. Hours apart.

Build to think. The loop is hours, not weeks.

Eric Ries, Build-Measure-Learn (The Lean Startup) · Teresa Torres, Continuous Discovery Habits
Personal story: needed a support channel fast for a team scaling from hundreds to tens of thousands of users. Instead of weeks of manual triage research, prototyped auto-triage, auto-tagging bugs vs features, clustering — now going to production. Teresa Torres' continuous discovery still holds. What changed is how fast you get to a useful conversation. Weeks to hours.

Belief #2
"Think in documents"

One-pagers, six-pagers, PRDs, design docs. The spec proved you understood the problem. Writing it was the thinking.

But a working prototype communicates intent at higher fidelity with fewer assumptions than any document can.

Before

Write spec → Review → Build

Weeks of alignment before code

Now

Idea → Prototype → Align

Show, don't tell

Think in prototypes.

IDEO, "Build to think" · Marty Cagan, "Inspired" · John Maeda, Design in Tech Report
Nuance: prototype when you need to learn. Spec when you need to align. A prototype doesn't replace a spec when the spec's function is getting 4 teams to commit.

"But a PM doesn't code"

You were always the thinker.

Now you're also the builder with AI.

taste judgment domain knowledge

These are the bottlenecks. Not coding.

Address the elephant head-on. PMs don't need to become engineers. AI handles implementation. Chat interfaces are enough to start — no IDE required. The bottleneck was never coding — it was always taste and judgment.
Belief #3
"Say no to most features"

When every feature costs months of engineering, "no" is your most important word.

When prototyping costs almost nothing, "no" prevents learning. Instead of debating X vs Y for weeks, prototype both and let users decide.

Cost of debate
is now greater than
Cost of validation

Prototype X

Prototype Y

↓ Run in parallel. Users decide.

Say no to bad problems, not experiments. When validation is cheap, run both.

Jeff Bezos, Amazon shareholder letters: "Double experiments, double inventiveness"
Each prototype informs the next. Each experiment makes the next cheaper. It compounds. But cheap experiments only work if they ladder to a clear strategy.
Belief #4
"AI is a tool you use"

Most treat AI like a search engine. Prompt, get answer, move on.

But when you capture team patterns as structured context, quality jumps. It goes from a tool to a teammate that compounds.

Boris Tane, "Context engineering is what makes AI magical" · Andrej Karpathy · LangChain
Demo: side-by-side - same prompt without context (generic) vs with instruction files. Kevin Weil (OpenAI) and Teresa Torres identify context engineering as critical. 3-4 min.
Belief #5
"Data-driven decisions"

"What does the data say?" became the default answer to every product question. Data was the tiebreaker, the decision-maker, the shield.

Data is essential for validation. But it can't tell you what to build next. That takes a point of view. The best PMs use both, but they start with judgment.

Before

Data → Decision

"What does the data say?"

Now

Judgment → Hypothesis → Data validates

"What do we believe? Let's test it."

Lead with judgment, verify with data.

Karri Saarinen, Linear · Lenny's Podcast
Linear doesn't A/B test their way to taste. They develop intuition, document principles, let judgment lead. Ronny Kohavi's research: ⅓ positive, ⅓ flat, ⅓ negative — data is a safety net for being wrong, not a compass for being right. Saarinen: "Opinionated software is about providing a clear direction and vision, rather than deferring every choice to the user."

What This Does NOT Mean

✕ Does NOT mean

  • PMs should replace engineers
  • Prototypes go to production unreviewed
  • Every PM must write code
  • Building replaces talking to customers
  • The spec is dead
  • Junior PMs should skip fundamentals

✓ DOES mean

  • PMs who build have new leverage
  • AI changes the economics of everything
  • Taste and judgment matter more
  • The idea-to-prototype gap is crossable
  • Every PM team needs building capability
  • Quality is the main job now
This slide earns credibility. The concern about AI replacing judgment-building work is real and worth acknowledging. Marty Cagan: AI tools without PM judgment is counterproductive.
